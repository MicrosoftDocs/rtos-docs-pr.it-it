# <a name="chapter-5---guix-display-drivers"></a>Capitolo 5 - Driver di visualizzazione GUIX

I driver GUIX Display definiscono l'interfaccia software tra l'area di disegno astratta e l'hardware di visualizzazione fisico. Il driver di visualizzazione GUIX implementa le funzioni di disegno di livello più basso che modificano effettivamente le informazioni sul colore dei pixel nella memoria canvas e trasferiscono la memoria del canvas nel buffer dei frame di visualizzazione fisici in sistemi a doppio buffer.

I driver di visualizzazione GUIX sono definiti da una struttura contenente i parametri di visualizzazione fisica e da un set di puntatori a funzione alle funzioni del driver di basso livello. Usando questi puntatori a funzione indiretti, le funzioni di disegno astratte canvas e widget vengono rese completamente indipendenti dai dettagli dell'hardware.

GUIX offre un set completo e completamente funzionale di funzioni di disegno predefinite per ogni profondità di colore e formato di colore supportato. Quando si implementa un driver di visualizzazione senza funzionalità di accelerazione hardware specifiche o altre considerazioni specifiche sull'hardware, queste funzioni di disegno predefinite sono in genere sufficienti per l'implementazione finale del driver. Per questi driver più semplici, l'unica funzione che normalmente deve essere implementata nel software driver è una funzione per configurare il dispositivo hardware. Questo comporta spesso l'inizializzazione di vari registri hardware per definire l'orologio del display LCD, le dimensioni dello schermo e così via. Per tutte le altre funzioni, l'implementazione del driver inizializza semplicemente GX_DISPLAY puntatori a funzione alle implementazioni di funzione predefinite per la profondità del colore e il formato desiderati.

Quando si implementa un driver di visualizzazione personalizzato, la procedura consigliata consiste nell'inizializzare prima di tutto i puntatori alle funzioni di disegno del driver di visualizzazione con l'implementazione software predefinita per la profondità del colore che si vuole supportare, quindi sostituire i puntatori a funzione dove si desidera chiamare le implementazioni della funzione personalizzata (se presenti). A tale scopo, è disponibile una funzione di configurazione predefinita per ogni profondità e formato di colore supportati. Ad esempio, se si scrive un driver video RGB in formato 5:6:5 a 16 bit, la prima operazione che il driver personalizzato normalmente fa è richiamare la routine di configurazione generica per questa profondità di colore:

UINT my_custom_565_display_driver(GX_DISPLAY *display)

```c
{
      
      // perform standard function pointer setup
      
      _gx_display_driver_565rgb_setup(display, GX_NULL,
      
             my_buffertoggle);

}
```

Il parametro my_buffer_toggle precedente è un puntatore alla funzione di attivazione/disattivazione del buffer del driver di visualizzazione (che può essere GX_NULL se il driver è a buffer singolo e disegna direttamente nel buffer dei frame hardware).

Se si scrive un driver di visualizzazione personalizzato, è necessario includere il file di intestazione gx_display.h nell'origine del driver personalizzato, che è un file di intestazione per uso interno non disponibile per il software a livello di applicazione.

Le funzioni di disegno a livello di visualizzazione GUIX ricevono come input un puntatore a **una GX_DRAW_CONTEXT** struttura . La **GX_DRAW_CONTEXT** definisce le coordinate di ritaglio per l'operazione di disegno corrente insieme al pennello e ai colori usati. Ogni funzione di disegno riceve come parametri aggiuntivi di input specifici per i requisiti della funzione.

La firma del punto **GX_DISPLAY** di ingresso del driver è definita come

```c
UINT <device>_graphics_driver_<format>(GX_DISPLAY *diplay)
```

Anche se il nome di questa funzione è interamente dell'implementatore, la convenzione per i driver forniti con GUIX è l'uso di un nome di dispositivo specifico dell'hardware nel campo e nel formato del colore per il campo <device> <format> precedente.
